# Estado da Task é derivado de timestamps, não de um campo explícito

Uma Task não tem campo `done` nem `status`: o estado é derivado de dois timestamps opcionais por `taskStatus(t)` — `completedAt` → `done`; senão `startedAt` → `on-going`; senão `todo`. Introduzimos o estado `on-going` (singleton global, a task em que o usuário trabalha agora) e consideramos um enum `status: "todo" | "ongoing" | "done"` e um aditivo `done: boolean` + `startedAt?`. Rejeitamos ambos porque manteriam um campo de estado redundante com os timestamps que a feature já exige (`startedAt` ordena a on-going no topo; `completedAt` alimenta o heatmap), e o modelo `done`+`completedAt` anterior obrigava manter os dois em sincronia à mão no toggle — uma invariante que some quando o estado passa a ser derivado. Custo aceito: migrator de store v3→4 que dropa `done` (zero transformação de dado, pois `done` ⇔ `completedAt` setado) e toda leitura de estado passa por `taskStatus()`.
